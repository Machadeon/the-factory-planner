### Findings
- [Skill check] Invoked `modern-web-guidance` and `frontend-design` per gate requirement ‚Äî both confirmed inapplicable (zero UI/frontend surface, pure TypeScript model/storage refactor, no components touched). No findings from either skill; documenting the check per instructions, not a defect.
- [Non-Goals / storage-service.ts] BLOCKING ‚Äî Non-Goals claims "No changes to `storage-service.ts` logic ... `loadLibrary`'s own branch structure ... unchanged." But `loadLibrary` (storage-service.ts:36-43) branches on `parsed.schemaVersion < CURRENT_SCHEMA_VERSION` to decide whether to call `migrateLibrary` at all. Spec R4 (storage-migrations) SHALLs "No deserialize, load, or migrate path SHALL branch on the runtime value of `schemaVersion`." This branch is exactly that. Design doesn't reconcile the contradiction ‚Äî it just declares the branch out of scope while the spec it's implementing forbids it.
- [Non-Goals / storage-service.ts, functional regression] BLOCKING ‚Äî once `CURRENT_SCHEMA_VERSION` is retired/pinned to `1` (per D3/R4), `loadLibrary`'s untouched condition becomes effectively `parsed.schemaVersion < 1`. Existing on-disk libraries at schema 1-4 (current `CURRENT_SCHEMA_VERSION = 5` in factory-storage.ts today) will evaluate `4 < 1` = false and **skip `migrateLibrary` entirely on load**, loading raw un-normalized data (no `folders: []`/`factories: []` structural defaulting from R5). This is a real functional regression the design doesn't surface as a risk, and it isn't covered by the "Risks/Trade-offs" section's existing schema-1‚Äì4-fidelity-loss risk (that risk assumes migration still runs and only degrades fields; this scenario is migration not running at all for a range of legacy inputs that used to trigger it).
- [D5 / module boundary] MAJOR ‚Äî D5 hedges on where `CURRENT_SCHEMA_VERSION` and `emptyLibrary` end up ("keeps or re-sources ... from `factory-storage.ts`, wherever they end up living") instead of committing. Current code (`app/models/factory-storage.ts:83,89-91`) defines both there and `storage-service.ts` imports both from there today. Spec R6 (storage-migrations) says `factory-storage.ts` SHALL contain **only** types, `serializeFactory`, `deserializeFactory`, `collectFactoryBundle`, `directDependencyIds`, `generateSlug`, `generateId` ‚Äî that "only" list omits both `CURRENT_SCHEMA_VERSION` and `emptyLibrary`. Either the spec's "only" enumeration is incomplete (should list these two) or the design must relocate them and D5 must say where ‚Äî as written, D5 and spec R6 are inconsistent and neither resolves it.
- [D3 / CURRENT_SCHEMA_VERSION fate] MAJOR ‚Äî D3 says "the field is already on the wire and every writer already sets it" (to `1`), implying `CURRENT_SCHEMA_VERSION` the exported constant still exists somewhere, presumably now `= 1`. But R4 (storage-migrations spec) says `schemaVersion` is written as literal `1` by `serializeFactory`/`emptyLibrary`/`migrateLibrary` ‚Äî doesn't say whether `CURRENT_SCHEMA_VERSION` survives as a named export or gets inlined. Given `useFactoryPageFlows.ts` and `storage-service.ts` both currently import `CURRENT_SCHEMA_VERSION`, D5's call-site update list should explicitly state whether that constant is kept (and where) or replaced with a literal `1` at each site ‚Äî currently unaddressed.
- [D2 / cycle-break console.warn volume] MINOR (carried context from spec review, now checking design treatment) ‚Äî Risks section accepts "Stub-mode's new `console.warn` volume during legitimate cyclic factory graphs" as a mitigation-by-acceptance, consistent with spec R3.S1's now-explicit warn-on-cycle-break behavior (verified in Pass 2 of spec-review.md). No new issue; design and spec agree here.
- [D1 anti-pattern check] none ‚Äî resolver-function-over-object-parameter is a reasonable seam (matches existing codebase convention of passing lookup closures, e.g. `partSlugLookup`/`recipeSlugLookup` module-level lookups); no anti-pattern.

The Migration Plan should at minimum acknowledge that `loadLibrary`'s branch needs to be re-examined, not asserted as a zero-risk revert.
- [Migration Plan / implementation order] MINOR ‚Äî step 4 ("Update remaining imports: `storage-service.ts`, hooks, `FactoryPage`") is where the `loadLibrary` branch issue above should have been caught; the plan treats `storage-service.ts` as an import-only touch throughout (matching the proposal's "Import-only touch" framing), but the cross-check above shows it isn't import-only once `CURRENT_SCHEMA_VERSION` semantics change. Proposal.md itself asserts "Its own logic (`loadLibrary`'s schema-version branch, consent gate, autosave keys) is unchanged" ‚Äî same unresolved contradiction traces back to the proposal, not just the design.

### Findings
- [D6 / R7 ‚Äî saveLibrary write-back cardinality unaddressed] MEDIUM, NEW ‚Äî D6 and R7 both specify that `migrateLibrary` now runs unconditionally on every load, but neither says whether the subsequent `saveLibrary(migrated)` write-back (storage-service.ts:41, currently only reached inside the old gated branch) also becomes unconditional. If `loadLibrary`'s `if` block structure is preserved as-is with only the condition swapped to "always true," every app load now does a synchronous `localStorage.setItem` write, even when nothing changed (R7.S2's no-op case) ‚Äî a new behavior (write-on-every-load instead of write-only-when-migrating) that isn't called out in Risks or in R7's scenarios. R7.S2 only asserts the *migrated library's shape* is equivalent to the input; it says nothing about whether a write occurs. This should either be an explicit accepted-risk bullet (extra localStorage write per load, cheap but real) or the design should specify skipping the save when the migrated result is referentially/structurally unchanged from the parsed input.
- [D6 ‚Äî no scenario for malformed/falsy schemaVersion] MINOR ‚Äî old code's condition was `!parsed?.schemaVersion || parsed.schemaVersion < CURRENT_SCHEMA_VERSION`, i.e. it had an explicit falsy-schemaVersion branch. D6's unconditional call trivially subsumes that case (any input, falsy or not, now gets migrated), so there's no functional gap ‚Äî but R7's two scenarios (legacy-numbered version, well-formed current version) don't explicitly exercise the missing/falsy `schemaVersion` case that the old code special-cased. Low risk since the unconditional call structurally covers it, but a third scenario would make the coverage airtight given this exact field was the source of the original bug.
