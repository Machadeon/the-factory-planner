### Findings
[D1] — Rejects a *new* context for transient UI state but doesn't address why `library`/`currentFactoryId` keep being prop-drilled from `LibraryDrawerSlot` through `LibraryTree`/`LibraryFolderRow`/`LibraryFactoryRow` instead of read via the *existing* `useLibraryContext()`. Confirmed: `FactoryPickerDialog`, `RecipeOptimizerPanel`, `FactoryOverviewComponent`, `ProductionTargetsBar`, and `LogisticsSection` all already consume `library`/`currentFactoryId` via `useLibraryContext()` rather than as props, and `LibraryDrawerSlot` already renders inside `<LibraryProvider>` in `FactoryPage.tsx` — so context is available at every level of the drawer tree today, unused. D1's rationale only rebuts the "new context for 6 transient state vars" strawman; it doesn't justify continuing to hand `library`/`currentFactoryId` down as props when every other consumer in this exact tree reaches them from context. This is an inconsistency with established codebase architecture, not addressed by the stated alternative-considered.
[D3 / Non-Goals] — `LibraryContextValue` (`app/contexts/LibraryContext.tsx`) exposes only `library`, `currentFactoryId`, `updatePartPointOverrides` — no CRUD mutators. The design adds seven new mutators to `useLibrary`'s return value (consumed today only via `libraryApi` prop) but never considers whether they belong on `LibraryContext` too, even though `LibraryContext` already wraps this exact component tree and other consumers reach `useLibrary`-sourced state through it. Scoping mutators out of context is defensible, but the design doesn't state it as a decision with a rejected alternative — it's simply absent, leaving future readers to wonder if this was considered.
[Risks/Trade-offs] — No `React.memo` (or equivalent) is proposed for `LibraryFolderRow`/`LibraryFactoryRow` despite D1 lifting all six transient state vars (`expandedFolders`, `editState`, `moveMenuFactory`, `menuState`, `deleteConfirmFactory`, `deleteConfirmFolder`) into `LibraryDrawer.tsx` and threading them down as props through every row at every depth. Any single state change (e.g. opening one row's actions menu) will re-render the entire tree, including all sibling/ancestor rows, since none are memoized. This codebase already uses `React.memo` for comparable list-row components (`RecipeOverrideRow.tsx`, confirmed via grep) — the design doesn't address why the same pattern isn't applied here, or note it as a deliberately deferred concern.
[D1 / Risks] — The six lifted state vars are threaded as six separate props at every recursion level of `LibraryFolderRow` (self-recursive, so props are re-passed at every depth) rather than grouped into a single object (e.g. `{ state, actions }`). Not called out as a decision or alternative; as prop count grows this is exactly the "prop drilling gets noisy" risk D1 already flags as a future concern, but no interim mitigation (e.g. a single bundled props object, which would also make `React.memo`'s shallow-compare in the finding above pointless without a custom comparator) is discussed.
[Migration Plan] — Step 2 ("extract components leaf-first... each commit keeping the app buildable") doesn't state whether these are separate commits within one PR, sequential PRs, or a single squashed commit. Given `tasks.md`/`review.md` gates in this repo's process operate at the PR level (per AGENTS.md's artifact table), the granularity of "each commit" matters for how `review.md`'s caveman-review pass will scope its diff — left ambiguous.
[Migration Plan / Rollback] — "No feature flag / rollback needed beyond normal git revert" is reasonable for a pure refactor, but doesn't address the specific risk this design itself calls out: if `React.memo` is added later (per the finding above) or the R3.S2 grep-diff misses a moved selector, a partial revert (e.g. only `LibraryFolderRow.tsx`) could leave `useLibrary`'s mutator surface out of sync with component expectations, since D6's extraction order makes rows depend on mutators existing first. The plan doesn't note that reverting the *component* extraction commits independently of the *mutator* commits could leave a broken intermediate state — worth one sentence given D6 explicitly sequences mutators-before-components.

### Findings
[D5a] — The requirement's justification for why `rowActions`'s `useMemo` deps are cheap/stable says: "deps: the setter functions, which are themselves `useState` setters and therefore already reference-stable." This is true for direct `useState` setters (`setEditState`, `setMenuState`, etc.) but D1a's own `rowActions` shape includes `toggleFolder`, which is not a `useState` setter — it's a plain function (`function toggleFolder(id) { setExpandedFolders(prev => ...) }`, current file lines 98-105) that closes over `setExpandedFolders`. As written today (and not addressed by D5a), a plain function declared in `LibraryDrawer.tsx`'s body is a new reference every render; D1a's own bundle also implies other handler-shaped entries beyond raw setters (`rowActions`'s listing in D1a ends with "..." suggesting more than the six named setters, and the current file has comparable handler functions like `commitRename` that a row needs). If `toggleFolder` (or any other handler-shaped member of `rowActions`) isn't itself `useCallback`-wrapped, then either: (a) it's included in `rowState`/`rowActions`'s `useMemo` deps array, in which case the memoized object changes every render anyway (since the dep is never stable), silently defeating D5a's whole point for exactly the reason D5a warns against; or (b) it's omitted from the deps array to keep memoization working, which is a stale-closure bug (ESLint's `react-hooks/exhaustive-deps` would flag it, and a fix-later `toggleFolder` bound to a stale `expandedFolders` setter would misbehave — though `useState` setters bound via updater-function form like `setExpandedFolders(prev => ...)` happen to be safe from staleness here, this is incidental, not something the design states as the reason it's safe). D5a should either require `toggleFolder`(and any other non-setter `rowActions` member) to be `useCallback`-wrapped too, or explicitly justify why leaving it out of the `useMemo` deps array is safe.
