## Context

`FactoryLibraryDrawer.tsx` (651 lines) is the last of the plan's "remaining big four" god components still in one file (`FactoryComponent`/`FactoryPage`, contexts, and model-type-safety already landed in prior phases per git history: `introduce-app-contexts`, `model-type-safety`). It renders a recursive folder/factory tree, owns six pieces of transient UI state in local `useState`, and calls `storage-service` functions directly followed by `saveLibrary` + `onLibraryChange(updated)` for every mutation — seven near-identical triples. `useLibrary.ts` already exists (from Phase 2) but today only exposes `replaceLibrary`/`reload`/`updatePartPointOverrides`; it has no CRUD mutators, so this is genuinely new hook surface, not a rename of existing code.

Constraints: no behavior change (E2E suite `tests/e2e/library/*.spec.ts` + `tests/integration/FactoryPage.test.tsx` + `tests/integration/contexts/library-nav-context.test.tsx` is the frozen contract — aria-labels/data-testids must not move); `LibraryDrawerSlot.tsx` is the only external call site and passes `libraryApi: ReturnType<typeof useLibrary>` already, so mutators are reachable without new prop plumbing at that boundary.

## Goals / Non-Goals

**Goals:**
- Six single-purpose files replace the one 651-line file, matching `plans/component-refactor.md` §3's `library/` tree.
- CRUD orchestration (mutate→persist→state-update) lives once, in `useLibrary`, not duplicated per call site.
- Bitwise-identical DOM output and interaction behavior (verified via the R3.S2 grep-diff and full test suite).

**Non-Goals:**
- No *new* context is introduced (that would exceed this phase's scope) — this change does adopt the *existing* `LibraryContext` for reads within the split tree (D1) since Phase 3 already landed it and it's already the established read path for sibling components; `libraryApi` (mutators) stays a prop per D3a.
- No fix to interrupted/overlapping-interaction edge cases (rename active + another menu opens) — today's behavior is preserved as-is, not audited or hardened (per spec R3).
- No change to `library-ops.ts` (import/merge) or `library-button-visibility` behavior.

## Decisions

**D1 — Prop-drilled transient state, not a new context, for the split; but `library`/`currentFactoryId` switch to the existing `LibraryContext`.** `expandedFolders`, `editState`, `moveMenuFactory`, `menuState`, `deleteConfirmFactory`, `deleteConfirmFolder` all stay as `useState` in `LibraryDrawer.tsx` and pass down as props, bundled into two objects (`rowState`, `rowActions` — see D1a) rather than six loose props, through `LibraryTree` → `LibraryFolderRow`/`LibraryFactoryRow` → `LibraryFactoryMenu`.
Alternative considered: a new `LibraryDrawerStateContext` to avoid prop drilling. Rejected — the prop chain is only 2-3 levels deep, and a new context for six single-drawer-scoped state vars is scope creep the proposal doesn't call for.

`library` and `currentFactoryId`, however, SHALL be read via `useLibraryContext()` directly, and independently, in every file that needs them, rather than passed down from a single top-level read: `LibraryDrawer.tsx` (root folders/factories, empty-state check), `LibraryFolderRow.tsx` (looking up `childFolders`/`childFactories` for its own `folder.id`), `LibraryFactoryRow.tsx` (`currentFactoryId` for the is-current highlight), and `MoveToFolderSelect.tsx` (`library.folders` for the dropdown options) each call `useLibraryContext()` themselves — none of them receive `library` or `currentFactoryId` as a prop from `LibraryDrawerSlot` or from a parent in the split tree. `LibraryDrawerSlot.tsx` already renders inside `<LibraryProvider>` (established in `FactoryPage.tsx`), and five existing components in this tree's sibling paths (`FactoryPickerDialog`, `RecipeOptimizerPanel`, `FactoryOverviewComponent`, `ProductionTargetsBar`, `LogisticsSection`) already consume `library`/`currentFactoryId` via context, not props — continuing to prop-drill them here would be inconsistent with established codebase architecture, not a neutral choice. `LibraryDrawerSlot.tsx`'s prop surface to `LibraryDrawer` shrinks to: `pinned`, `libraryApi` (for mutators — see D3a), `flows` (load/new/import handlers), `onPinChange`.
Alternative considered: keep `library`/`currentFactoryId` as props for symmetry with the mutator-bearing `libraryApi` prop. Rejected — mutators and reads are different concerns; reads already have an established context path in this exact component tree, mutators don't (D3a).

**D1a/D1b — One authoritative table, derived exhaustively from every function/value `FactoryLibraryDrawer.tsx` defines today, is the single source of truth for `rowActions` membership, prop contracts, and memoization stability rules.** Prior review passes (3, 4, 5) each caught one missing member of `rowActions` added incrementally, and a restated-elsewhere list drifted from the table meant to correct it. Fix: state this exactly once, here, and every other decision (D5a's memoization, D1's context split) references this table instead of re-listing members.

Every non-JSX-producing function/derived-value in today's `FactoryLibraryDrawer.tsx`, where it lands post-split, and its `React.memo`/`useCallback` stability rule:

| `toggleFolder(id)` | `rowActions.toggleFolder` | `useCallback`, empty deps |

`rowActions`' complete, closed member list per this table: `{ toggleFolder, commitRename, closeMenu, handleAddFolder, setEditState, setMoveMenuFactory, setMenuState, setDeleteConfirmFactory, setDeleteConfirmFolder }` — 9 members, no others. `LibraryFolderRow`/`LibraryFactoryRow` each receive `rowState` and `rowActions` as their only state-related props (bundled into two objects, not nine-plus individual props, since `LibraryFolderRow` re-passes these at every recursion depth).

`LibraryFactoryMenu.tsx`'s prop contract: rendered exactly once, in `LibraryDrawer.tsx` (matching today's single shared `<Menu anchorEl={menuState?.anchorEl}>` — not once per row). Receives: `menuState: MenuState | null`, `onClose` (= `rowActions.closeMenu`), `onRename` (= `rowActions.setEditState`, called with `{ type: "factory", id }`), `onMove` (= `rowActions.setMoveMenuFactory`), `deleteConfirmFactory`/`onDeleteConfirmFactoryChange` (= `rowState.deleteConfirmFactory`/`rowActions.setDeleteConfirmFactory` — it needs both, since it owns the `ConfirmDialog` JSX itself, not just the trigger), and `libraryApi: Pick<ReturnType<typeof useLibrary>, "duplicateFactory" | "deleteFactory">` (the table's `handleDeleteFactory` row requires `libraryApi.deleteFactory` be reachable here, alongside `duplicateFactory` for the Duplicate menu item — both are on the `Pick`). It does NOT receive the full `rowState`/`rowActions` objects. It owns the `deleteConfirmFactory` `ConfirmDialog` JSX (moved from `LibraryDrawer` per R1.S1's one-purpose-per-file rule); `deleteConfirmFactory`/`setDeleteConfirmFactory` state ownership stays in `LibraryDrawer.tsx`'s `rowState`/`rowActions` per R1.S3 — only the JSX moves, hence both are passed down as props here.

`MoveToFolderSelect.tsx`'s prop contract (the file's only remaining unspecified one): rendered inline by `LibraryFactoryRow.tsx` when `rowState.moveMenuFactory === factory.id`. Receives: `factory: SerializedFactory` (for `factory.id`/`factory.folderId`, the current value), `folders: FactoryFolder[]` (the dropdown options — read via its own `useLibraryContext()` per D1, not passed down, consistent with every other read in this tree), and `libraryApi: Pick<ReturnType<typeof useLibrary>, "moveFactory">` plus `onMoved: () => void` (= `rowActions.setMoveMenuFactory` called with `null`, fired from `onChange` after `libraryApi.moveFactory` per the table's `handleMoveFactory` row). Not memoized — see D5a.

Alternative considered (for the two-objects-not-nine-props question): nine separate props (matches today's implicit closures 1:1). Rejected — nine props re-threaded at every recursive call is exactly the noise this design must not introduce; two grouped, `useMemo`-stabilized objects is a smaller, stable diff surface.

**D2 — `LibraryFolderRow` self-recurses; `LibraryTree` is not a recursive component.** `LibraryTree` renders the sorted root-level folders/factories once. `LibraryFolderRow` imports itself and calls itself for `childFolders`, matching today's `renderFolderRow(f, depth + 1)` recursion pattern exactly (same `depth` prop threading for indentation).
Alternative considered: hoist recursion into `LibraryTree` with a `depth` param and have it call itself, with `LibraryFolderRow` staying "pure" (no self-import). Rejected — that would make `LibraryTree` respond to two different questions ("what's the root list" and "how do I recurse a folder's children"), reintroducing a two-purpose file, which is what R1 exists to prevent.

**D3a — Mutators stay on `useLibrary`/`libraryApi`, not on `LibraryContext`.** The seven new CRUD mutators are added to `useLibrary`'s return value only; `LibraryContextValue` (`library`, `currentFactoryId`, `updatePartPointOverrides`) is not extended.
Alternative considered: also expose the mutators via `LibraryContext` so any consumer in the tree could reach them without a prop. Rejected — `LibraryContext` today carries only read state plus the one mutator (`updatePartPointOverrides`) that many unrelated consumers need; the seven library-tree-specific CRUD mutators have exactly one consumer (the library drawer tree), so routing them through a prop (`libraryApi`, already threaded by `LibraryDrawerSlot` today) is simpler than growing a shared context's surface for single-consumer needs. If a second consumer emerges later, promoting them to context is a small follow-up, not a blocker here.

**D3 — Mutators wrap existing `storage-service` functions 1:1; no new `storage-service` exports; each mutator SHALL be `useCallback`-wrapped.** `useLibrary.renameFactory(id, name)` = `updateFactory(library, { ...factory, name })` (existing `updateFactory` export, previously unused by the drawer — the drawer inlined an equivalent `.map` instead). All other mutators wrap the storage-service function of the same name that already exists (`addFactory`, `removeFactory`, `addFolder`, `renameFolder`, `removeFolder`, `moveFactory`). Every mutator SHALL be defined with `useCallback` (matching `useLibrary.ts`'s existing `replaceLibrary`/`updatePartPointOverrides` pattern), giving them stable identity across renders — this is a prerequisite for D5a's `rowActions` memoization (`commitRename`'s `useCallback` deps include `libraryApi.renameFactory`/`renameFolder`, which only avoids spurious churn if those are themselves stable).
Alternative considered: add a new `storage-service.renameFactory` export mirroring `renameFolder`, and have the hook call that. Rejected in spec-review pass 1 as scope creep beyond the proposal's declared Impact (`useLibrary.ts` + `LibraryDrawerSlot.tsx` only) — `updateFactory` already does the job.

**D4 — `duplicateFactory` mutator owns id/timestamp/name-suffix construction; callers pass the untouched source.** Moves `handleDuplicateFactory`'s body (id via `generateId()`, `" (copy)"` suffix, fresh `createdAt`/`updatedAt`) into the `useLibrary` mutator.
Alternative considered: keep copy-construction in the component and have the mutator take a pre-built `SerializedFactory`. Rejected — that leaves persistence-adjacent business logic (what counts as "a duplicate") outside `useLibrary`, contradicting R2's intent that components only call mutators, never reason about library semantics.

**D5 — `addFolder` mutator returns `{ folder }` (mirrors `storage-service.addFolder`'s shape) so the caller can chain UI-only reactions.** Today's `handleAddFolder` does three things after creating the folder: persist, expand the parent, and enter rename-edit-state. Only the first is inherently a mutator's job — the other two are UI decisions specific to `LibraryDrawer`'s own `expandedFolders`/`editState`. The mutator returns the created folder; `LibraryDrawer`'s `handleAddFolder` wrapper (not the hook) does `const { folder } = libraryApi.addFolder(...); setExpandedFolders(...); setEditState(...)`.
Alternative considered: mutator takes an `onCreated` callback. Rejected — unnecessary indirection when a return value works and matches `storage-service.addFolder`'s existing `{ lib, folder }` convention.

**D5a — `LibraryFolderRow`, `LibraryFactoryRow`, and `LibraryFactoryMenu` are wrapped in `React.memo`; `rowState`/`rowActions`/each `rowActions` member's stability is a hard requirement per D1b's table, not an example.** Since D1a/D1b lift all transient state to `LibraryDrawer.tsx` and thread it down as `rowState`/`rowActions`, an unmemoized row tree would re-render every row (all depths, all siblings) on any single state change (e.g. opening one row's actions menu). This codebase already establishes the precedent for list rows (`RecipeOverrideRow.tsx` uses `React.memo`). `MoveToFolderSelect.tsx` is the one split file NOT memoized — it's only ever mounted conditionally (`isMoving && <MoveToFolderSelect .../>`), so there's no sibling-row re-render cost to avoid.

`React.memo`'s shallow prop comparison only pays off if object props are stable across renders where their contents didn't change — this is a requirement, not a nice-to-have. D1b's table is the complete, authoritative list of every `rowActions` member and its exact stability rule (which `useCallback` deps, or "raw setter, no wrapping"); nothing here restates it. Of `LibraryFactoryMenu`'s props: `onClose`/`onRename`/`onMove`/`onDeleteConfirmFactoryChange` are `rowActions` members already covered by that table; `libraryApi` is stable per D3; `deleteConfirmFactory` is a raw value (not a callback) drawn from `rowState`, itself `useMemo`-stabilized per D1b's `rowState` row — so it changes reference only when its underlying value changes, same as every other `rowState` consumer.

Without `rowState`/`rowActions` `useMemo`-wrapped at the top (per D1b's table) and every non-setter member `useCallback`-wrapped exactly as D1b's table specifies, `rowState`/`rowActions` are new object literals every render, `React.memo`'s shallow compare always sees "changed" props, and this decision is silently defeated — implementers MUST NOT skip any row of D1b's stability column and MUST NOT treat `React.memo` alone as sufficient.
Alternative considered: no memoization, matching the pre-split component's behavior (the whole tree re-rendered together as a single component already, so this isn't a regression in absolute terms). Rejected — the split makes per-row re-render *visible* as a maintenance property others will read as a signal for how rows in this codebase are supposed to behave; matching the `RecipeOverrideRow` precedent costs one `React.memo()` wrapper per row file plus two `useMemo` calls in the parent.

**D6 — Extraction order: leaf-first.** Implementation proceeds `MoveToFolderSelect` → `LibraryFactoryMenu` → `LibraryFactoryRow` → `LibraryFolderRow` → `LibraryTree` → `LibraryDrawer`, with `useLibrary` mutators added before any component extraction (so rows are written against the final mutator API, never against the old inline calls). This lets each step be verified against the running app/tests incrementally rather than one big-bang file move.

## Risks / Trade-offs

[Prop-drilling `rowState`/`rowActions` through `LibraryTree` even though `LibraryTree` doesn't read them] → Acceptable per D1a/D1b's bundling (two objects — `rowState`'s 6 values and `rowActions`' 9 members — not 15 loose props) and mitigated by D5a's `React.memo`; matches existing codebase patterns (`ProductionLineComponent` → `AssemblyLineComponent` prop threading predates this change). If it gets noisy, promoting `rowState`/`rowActions` to a small drawer-scoped context is a future, non-blocking follow-up.

[`LibraryFolderRow` self-import creates a small circular-looking dependency graph] → Not actually circular (a component importing its own file to recurse is standard React); flagged here only so the design-review pass doesn't re-raise it as a novel risk.

[Manual selector diff (R3.S2 grep) could miss a selector that moved to a JS template literal in a way grep's simple patterns don't catch] → Full E2E suite (Playwright, real DOM assertions via `getByRole`/`getByTestId`) is the actual safety net; the grep is a fast pre-check, not the sole verification.

## Migration Plan

This change lands as a single PR (per this repo's `spec-driven-reviewed` process — one `review.md` gate covers the full diff), landed as separate commits within that PR for reviewability:

1. Add mutators to `useLibrary.ts` (D3, D3a, D4, D5), with unit tests (`tests/unit/hooks/useLibrary.test.ts`) — one commit, before any component touches them.
2. Extract components leaf-first (D6: `MoveToFolderSelect` → `LibraryFactoryMenu` → `LibraryFactoryRow` → `LibraryFolderRow` → `LibraryTree` → `LibraryDrawer`) — one commit per file, each keeping the app buildable; `FactoryLibraryDrawer.tsx` remains the live export consumed by `LibraryDrawerSlot` until the last extraction commit lands.
3. Swap `LibraryDrawerSlot.tsx`'s import from `FactoryLibraryDrawer` to `library/LibraryDrawer` and delete `FactoryLibraryDrawer.tsx` — one commit.
4. Run full suite (`npm run test:run`, `npm run test:e2e`, `npm run build`) before marking tasks complete.

No feature flag / rollback needed beyond normal git revert — this is a pure refactor behind no runtime toggle. Because step 2's commits build components against the mutator API added in step 1 (D6's mutators-before-components sequencing), a partial revert of only a later component-extraction commit (leaving step 1's mutator commit in place) is safe — mutators have no component-side dependencies. Reverting step 1's mutator commit *without* also reverting every step-2 commit that consumes it is not safe and should not be done in isolation; revert step 2 commits first (in reverse order) if step 1 needs to be undone.
