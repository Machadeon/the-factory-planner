### Findings
- [overview-sidebar-structure R1] — doesn't name the existing consumer of `FactoryOverviewComponent` (`app/components/factory/FactorySidebar.tsx`) that must be updated to import `OverviewSidebar`; proposal.md Impact mentions "any import site" vaguely but the spec requirement never states this update site explicitly or makes it testable (no scenario checks the import site was updated).
- [overview-sidebar-structure R1] — "Power & Modules section using `PowerSummary`" is not listed as its own named file in the six sections, but R2.S1 later requires the directory to contain `PowerSummary.tsx` as a file; R1's prose never says which file renders the Power & Modules section container (only R3 implies `PowerSummary` is the power row, not the whole section) — ambiguous whether Power & Modules gets its own `PowerModulesSection.tsx` or is inlined in `OverviewSidebar.tsx`.
- [overview-sidebar-structure R1.S2] — untestable as written: "the corresponding section and its preceding divider are not rendered" doesn't specify how a test observes "not rendered" (no aria-label/testid referenced) — should reference a concrete selector or absence assertion.
- [overview-sidebar-structure R2.S1] — "none of these files import rendering logic from another section file" is not mechanically testable (no lint rule or test named to enforce it); as written this is an aspirational code-review criterion, not a verifiable requirement/scenario.
- [overview-sidebar-structure R3] — scope creep vs proposal: proposal.md "What Changes" says `PowerSummary.tsx` is "consumed by both the overview's Power & Modules section and `AssemblyLineControls.tsx`", but doesn't mention Power Shards / Somersloops rows (also currently inline in `FactoryOverviewComponent.tsx` lines 247-258) needing extraction — spec R2.S1 requires `PowerSummary.tsx` to exist but never clarifies scope is the power-only row vs whole Power & Modules block, risking either under- or over-scoped implementation.
- [overview-sidebar-structure R3.S1] — format string doesn't match source exactly: source renders avg and min-max in two separate `<span>` elements (lines 233-240), not a single `"{avg} MW avg · {min}–{max} MW"` string — spec should say "renders equivalent to" or specify the two-span DOM structure if DOM parity (R4.S2) is meant to be frozen.
- [overview-sidebar-structure R4] — "Component prop contracts MAY change only to route data through new file boundaries" is vague/untestable — doesn't enumerate which prop signatures are allowed to change (e.g., does `PartRateSummary` keep identical props, or do `OutputsSection`/`InputsSection` get new required props like `factory`/`rateLookup`?). No scenario pins this down.
- [overview-sidebar-structure R4.S2] — "rendered text content, icons, and allocation-bar percentages are identical" has no defined mechanism (no visual regression tool referenced in this codebase per AGENTS.md — testing stack is vitest/RTL/Playwright, not screenshot diffing) — likely means integration test assertions, but the requirement doesn't say that, making it hard to know what "verify" means operationally.
- [overview-sidebar-structure] — spec never addresses the existing `onRebuild?: () => void` prop on `FactoryOverviewComponent` (currently unused per source comment line 27-28) — proposal doesn't mention whether `OverviewSidebar` keeps, drops, or renames this prop; a dropped/renamed prop is a prop-contract change not covered by any requirement.
- [overview-sidebar-structure] — no requirement/scenario addresses `tests/integration/FactoryOverviewComponent.test.tsx` or `tests/e2e/overview/sidebar-section-visibility.spec.ts` (both exist today per repo, both reference `FactoryOverviewComponent` naming/behavior) — R4 requires suites to pass but doesn't say these specific test files get renamed/relocated to match the new `overview/` structure, leaving ambiguous whether stale filenames referencing the deleted component are acceptable.
- [consumer-links R1] — proposal.md's "What Changes" explicitly says hoist target is `app/models/consumer-links.ts`, but AGENTS.md's documented directory convention only describes `app/models/` for domain model classes (`factory.ts`, `production-line.ts`, etc.) with kebab-case naming for "Model/service files" — spec doesn't justify why a pure function module belongs in `app/models/` vs e.g. co-located with `graph-model.ts`'s siblings; not itself wrong, but the design rationale is undocumented and no scenario tests the file's location convention consistency.
- [consumer-links R2] — description of "SHALL return an empty map when library or currentFactoryId is absent" duplicated near-verbatim as R2.S1 scenario — R2's prose and R2.S1 scenario say the same thing with no added test detail (e.g. what about `currentFactoryId` being an empty string `""`, which is falsy but not null/undefined — behavior per source `if (!currentFactoryId)` would also return empty map, but requirement text says "null/undefined" only, an inconsistency with actual guard code which is a truthiness check, not a null/undefined check).
- [consumer-links R2.S2] — edge case at exactly `net === RATE_EPSILON` is untested: source code excludes when `net <= RATE_EPSILON` (source line 277 in graph-model.ts, line 66 in FactoryOverviewComponent.tsx), so the boundary itself is exclusive, but the spec's Scenario R2.S2 only says "at or below" without a concrete boundary-value scenario/example to pin the off-by-one behavior for a reviewer/implementer.
- [consumer-links R2] — doesn't mention that the graph-model.ts version iterates `factory.allOutputs(): Part[]` (keyed by `.slug`) while the FactoryOverviewComponent inline version iterates `factoryOutputs` (i.e., `FactoryOutput[]` from `getOutputInfo()`, keyed by `.part.slug`) — functionally equivalent today (verified: `getOutputInfo()` maps over `allOutputs()`), but the spec's single unified signature `deriveConsumers(factory, {library, currentFactoryId})` returning by output part doesn't state which of the two existing call sites' output-ordering/duplicate-key behavior is authoritative if they ever diverge — low risk but worth one sentence given this is literally the reason for the hoist.
- [consumer-links] — no requirement covers what happens to the unit test coverage proposal.md promises ("new unit test for consumer-links.ts if none exists") — spec.md itself has zero scenarios asserting test-file existence/location, leaving that solely to tasks.md; acceptable if tasks.md picks it up, but flagging since proposal explicitly calls it out as an open question ("if none exists").
