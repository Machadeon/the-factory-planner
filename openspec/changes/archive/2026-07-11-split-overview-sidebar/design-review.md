### Findings
- [Decision 3] 🔴 bug: leaf sections call `useFactory()` (mutable proxy) with no props and no mention of `useFactoryUpdateSubscription()`. Per `app/contexts/FactoryContext.tsx`'s own doc comment, `useFactory()` is "Never render off this" — reactivity requires `useFactoryUpdateSubscription()`/`useFactorySnapshot()`. Every existing top-level panel (`FactoryOverviewComponent`, `PlanningSection`, `LogisticsSection`, `OptimizationSection`) calls `useFactoryUpdateSubscription()` exactly once at its root — none call it per-leaf. Design doesn't say whether each of the 5 new leaf sections independently calls `useFactoryUpdateSubscription()` (redundant re-render wiring, 5x subscriptions to the same `rateLookup`) or whether `OverviewSidebar` keeps the single call and leaves silently rely on parent re-render without their own snapshot read (in which case `useFactory()` alone in a leaf won't re-render that leaf on a mutation scoped elsewhere in the tree, e.g. a mutation that doesn't touch the parent's read but does touch a value the leaf itself reads/renders). Pin down: which component(s) call `useFactoryUpdateSubscription()`, and confirm leaves that read `factory.<field>` for render output use `useFactorySnapshot()` not raw `useFactory()`.
- [Decision 3] 🟡 risk: "sections read context directly; no new prop interfaces" conflicts with this being harder to unit-test in isolation — every leaf section becomes untestable without a full `FactoryProvider`/`LibraryProvider`/`NavigationProvider` wrapper, whereas today's single `FactoryOverviewComponent` already requires that setup once. Not fatal (same testing cost either way, just multiplied per file) but design doesn't acknowledge the tradeoff or reference `tests/integration/overview/`'s planned wrapper setup for the 5 new leaf test files (spec R4.S1 only renames one test file, doesn't mention per-section test files).
- [step 3, Decisions] 🔵 nit: "lifting the corresponding JSX block out of `FactoryOverviewComponent.tsx` verbatim" for `ConsumersSection` is inconsistent with Decision 4, which says the memo body changes (`inline loop` → calls hoisted `deriveConsumers`) — that's not verbatim, it's a real logic edit bundled into "verbatim" extraction. Contradicts the extraction-order framing that steps 1-3 are copy-only.
- [Risks/Trade-offs, `PowerSummary` risk] ❓ q: mitigation says "surrounding layout ... stays owned by each call site" — but doesn't specify `PowerSummary`'s own root element (span vs div, flex vs inline) or confirm it won't introduce a wrapping element that breaks `AssemblyLineControls`'s `flex flex-row gap-x-1` container (source line 184) vs the overview's `flex flex-row items-center gap-x-2 mb-1` container (source line 226) — two different flex configs wrapping the same shared component. Needs one sentence confirming `PowerSummary` renders as a fragment/inline group with no own wrapping box that would fight either parent's flex/gap.
- [Migration Plan] 🟡 risk: "each is independently revertable via `git revert`" is asserted but the actual task/commit granularity isn't defined here (design.md, unlike tasks.md, doesn't commit to one-commit-per-step) — if steps 1-8 land as a single PR/commit (explicitly stated: "Single-PR, in-repo refactor"), the revert-per-step claim is aspirational, not actually available; a single-PR revert reverts everything, not step-by-step. Either commit to per-step commits within the PR or drop the per-step revertability claim.
- [Migration Plan] — no rollback consideration beyond CI gates for a change that deletes a 343-line file and touches a second unrelated file (`AssemblyLineControls.tsx`). If `npm run build` passes but a runtime-only issue surfaces post-merge (e.g. a context read ordering issue per the finding above), the design doesn't note a feature-flag or staged-rollout option — reasonable for an internal-tool single-PR refactor, but "no deploy/rollback concerns" understates risk given the `useFactoryUpdateSubscription` ambiguity above; if that ambiguity resolves to "silent stale render," it will not be caught by `npm run build` or likely by the specified tests since existing integration tests probably don't assert on cross-leaf re-render timing.
- [Decisions, general] — design.md doesn't address `AGENTS.md`'s "Props interfaces: `XxxProps` suffix" convention at all for the leaf sections that take zero props — reasonable since no props exist, but worth one line confirming no empty/unused `XxxSectionProps` interface is scaffolded out of habit (a common cleanup miss).
- [Context: modern-web-guidance] — could not load `modern-web-guidance` skill; it is not present in this environment's available skill list (only `example-skills:frontend-design` and other unrelated skills are available). Skipping modern-web-standards-specific findings rather than fabricating them against an unavailable rubric; flagging the gap so the coordinator can supply the skill or confirm it's out of scope for this repo.

- [Architecture/AGENTS.md conventions] — no conflicts found. ### Findings
- [Decision 3, "Reactivity wiring" paragraph] 🟡 risk: claims leaf sections calling `useFactorySnapshot()` "mirrors" the established `PlanningSection`/`LogisticsSection`/`OptimizationSection` pattern, but it doesn't — all three actual precedents call `const factory = useFactory(); useFactoryUpdateSubscription();` together at the panel root and render off the **mutable proxy** (`useFactory()`), relying on the subscription hook to trigger re-renders; none of them call `useFactorySnapshot()` directly. The new leaf-level `useFactorySnapshot()` calls are a different (also valid, per `FactoryContext.tsx`'s own comment: "Leaf-level scoping uses `useSnapshot(subObject)` directly instead") pattern, not the same one. Functionally fine — `useFactorySnapshot()` in a leaf gives that leaf its own independent, correctly-scoped valtio subscription — but the design's framing overstates the parity with precedent. One sentence should clarify these are two different reactivity strategies (subtree-subscription-at-root vs per-leaf-snapshot), both valid, rather than implying the leaves match the three existing panels' approach.
- [Decision 3 / Reactivity wiring] ❓ q: if every leaf independently calls `useFactorySnapshot()` (its own `useSnapshot` subscription), is `OverviewSidebar`'s root-level `useFactoryUpdateSubscription()` call still doing useful work, or is it now redundant? `useFactorySnapshot()`-per-leaf already subscribes each leaf to whichever fields it reads, which is finer-grained than the root's blanket `rateLookup` subscription. Not a bug, but design.md doesn't explain why both layers of subscription are kept — worth one sentence (e.g. "root subscription is a safety net for values no leaf snapshot-reads directly, e.g. derived-but-unread fields" or else drop the root call as unnecessary).
- [frontend-design] — none, as expected. Re-confirmed non-goal "No visual redesign" still holds; `PowerSummary`'s new fragment-based (no wrapping box) decision doesn't introduce any visual/DOM change beyond what R3.S1 already specifies. No findings from this skill.
- [modern-web-guidance] — still unavailable in this environment; skipped per coordinator direction, not treated as a finding against design.md.

### Remaining scope
Both remaining items are precision/framing issues in the "Reactivity wiring" paragraph, not functional bugs — the actual behavior described (root subscribes via `useFactoryUpdateSubscription()`, leaves render off `useFactorySnapshot()`, `SuppliersSection` uses `useFactory()` only for its one write) is internally consistent and will work correctly. The gap is that the paragraph's claim of "exactly matching" the three existing panels' pattern is inaccurate (they use `useFactory()`, not `useFactorySnapshot()`, for reads), and the design doesn't explain whether the root-level subscription is still load-bearing once every leaf has its own snapshot subscription. Both are one-sentence fixes. Status remains CONCERNS until the mismatch-with-precedent claim is corrected or justified.
