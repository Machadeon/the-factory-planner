### Findings
[Decision 1 — thin-wrapper vs inline] — Justification is test-preservation only ("tie-breaker"), not an architectural one; the design admits "no other Factory mutator in this file delegates to a child-model pure-computation method" — i.e. this introduces a new pattern shape into factory.ts with a single-instance precedent. Acceptable as a pragmatic call but the design doesn't consider the alternative of updating `production-line.test.ts` to test the math via the new `Factory.splitRecipeRates` call instead (which `removeAssemblyLine`'s own tests likely do through `Factory`, not the child model) — that path was never evaluated, only "delete the child method" vs "keep it," so the alternatives analysis is incomplete.
[Decision 1] — `ProductionLine.splitRecipeRates()` stays public after this change but becomes dead code from any component's perspective (only called by the new `Factory.splitRecipeRates` and the existing unit test). A public method that exists solely to be unit-tested and wrapped is a code smell the design doesn't name or defend beyond "remains a public method (not deleted)" — no comment/doc-string guidance proposed to mark it as "call through Factory.splitRecipeRates" for future readers, so the mutation-contract violation this whole change fixes could recur if a future component reaches for `ProductionLine.splitRecipeRates()` directly again, since nothing besides the new regex test signals not to.
[Decision 2 — rejectAssembly(recipe) signature] — The stated justification ("passing `pl` would be an unused parameter") is syntactic, not semantic, and is arguably the wrong justification even though the conclusion may be right: the real reason `pl` is unneeded is that reject state lives in the *global* `factory.optimizer.enabledRecipes` keyed by recipe slug, unlike `acceptAssembly(pl, recipe)` which mutates a specific `AssemblyLine.autoCreated` flag and genuinely needs `pl` to disambiguate which line's assembly-line instance to flip. Confirmed via factory.ts:643-646 (`acceptAssembly` finds `al` by scanning `pl.assemblyLines`). Design.md never states this semantic distinction, so a future reader comparing `acceptAssembly(pl, recipe)` to `rejectAssembly(recipe)` has no explanation for the asymmetry beyond "lint would complain," which invites someone to "fix" the asymmetry incorrectly later (e.g. by adding `pl` back for symmetry, silently breaking nothing today but obscuring the real reason two accept/reject twins have different arity).
[Decision 2 vs proposal/spec] — Confirmed this deviates from the proposal (`Factory.rejectAssembly(pl: ProductionLine, recipe: AnyRecipe)`, line 9) and the approved spec (R5.S9's `Factory.rejectAssembly(pl, recipe)`). Design.md asserts this is "an implementation-level signature detail, not a behavior change the spec pins," which is defensible for R5.S9 specifically (recompute semantics, not arity) — but the proposal's "What Changes" and "Impact" sections both explicitly commit to the two-arg signature in prose. Design.md doesn't flag that the proposal itself will read as stale/inaccurate once implemented with the one-arg signature, nor propose a proposal.md correction. Since proposal.md is not re-opened after design in this workflow (only specs are the enforced contract), this leaves a permanent discrepancy between what proposal.md says was built and what actually ships — worth a one-line note in design.md acknowledging the proposal text will be superseded, or a task to amend proposal.md.
[Non-Goals / Open Questions — onRejectChoice deferral] — The deferred `applyRejectChoice` gap (lines 152, 162) is the same escape class the widened R4.S4 test is meant to close, and it sits in the exact same component, in functions adjacent to the ones being fixed (`rejectLine`/`rejectAssembly` immediately above `onRejectChoice` in the file). Design.md's own regex (Decision 3) bans only `applyRejectSilent(`, not `applyRejectChoice(`, so after this change ships, `tests/unit/mutation-contract.test.ts` will still pass while `onRejectChoice()` leaves `rateLookup`/index state stale on the prompted-reject path — i.e., the mutation contract is still violated in shipped code, just now by a different helper name the test doesn't check. The design frames this as "mirrors this issue's own history" (flag now, fix later) without acknowledging the irony: this is the second time the exact same gap is being knowingly shipped with a promise to file a follow-up, in the same file, in the same review cycle that exists to prevent that pattern. A stronger design would at minimum widen R4.S4's regex to also ban `applyRejectChoice(` now (cheap, same shape, closes the whole escape class this change targets) even if the actual `Factory.rejectLine`/`rejectAssembly`-style wrapper for the prompted path is deferred — as written, the "fix" leaves a known-identical hole in the exact mechanism (the test) that's supposed to prevent recurrence.
[Risks/Trade-offs] — The `onRejectChoice` risk entry says mitigation is "file as a follow-up GitHub issue before archive," which satisfies AGENTS.md's archive gate procedurally, but doesn't address the substantive risk (rateLookup staleness reachable today via the prompted-reject UI path, likely the *more common* path since `shouldPromptReject` gates on user settings and silent-reject is the exception case). No severity assessment of how often the prompted path is hit relative to the silent path — if prompted-reject is the default/common flow, "no immediate resolution" undersells how live the underlying bug (issue #23's actual symptom class) remains post-fix.
[Decision 4 — regression test] — Says a rate-based assertion isn't possible for `rejectLine`/`rejectAssembly` ("`enabledRecipes` isn't part of `rateLookup`") and falls back to a spy on `_updateRates`. Reasonable, but the design doesn't specify what the spy asserts beyond "proving `_updateRates` runs" — doesn't call out that a spy-based test here is a weaker regression guard than R5.S7's rateLookup-value assertion (per R7's stated preference for "concrete values, not... captured pre-change snapshot"), and doesn't explain why a stronger assertion (e.g., asserting `optimizer.enabledRecipes` changed, since that's directly observable state) isn't used instead of/alongside the spy. A spy proves the recompute *was called*, not that state is *actually consistent* afterward — thinner than what R2's intent (derived state left consistent) calls for.
[Migration Plan] — "None" is accurate; verified no schema/serialized-field changes (CURRENT_SCHEMA_VERSION unaffected, no new persisted fields). No issue here.
